# サンプルドキュメント

このファイルはチャンク分割のテストに使用する。

## 背景

Megurip アプリケーションのインフラストラクチャを Terraform で管理するリポジトリである。
AWS のサーバーレスアーキテクチャ（Lambda, API Gateway, S3, CloudFront）を活用し、
コスト効率の高いインフラ構成を実現している。バックエンドは Go/Echo で実装され、
aws-lambda-go-api-proxy を使用して Lambda 上で動作する。Worker Lambda は
Python 3.13 で実装され、ルート計算や AI 最適化といった非同期処理を担当する。
データベースには Neon PostgreSQL を採用し、サーバーレスとの相性を最大限に活かしている。

## 設計方針

レゴブロック型アーキテクチャを採用する。modules/ がレゴブロック（部品）で、
environments/ が組み立て場所という明確な責務分離を実現している。
各モジュールは単一責任の原則に従い、他のモジュールの内部実装に依存しない。
モジュール間の連携は environments/ レイヤーで変数の受け渡しによって行われ、
モジュール自体は独立してテスト可能である。この設計により、新しい環境の追加や
既存環境の変更が容易になり、インフラの変更に対するリスクを最小限に抑えている。

## コードブロック内のテスト

以下のコードブロック内の `##` で分割してはいけない。コードブロックの検出は
バッククォート3つの開始・終了をトラッキングすることで実現する。
ネストされたコードブロックや異なる言語指定にも対応する必要がある。

```markdown
## これは分割しない
この行もコードブロック内なので無視する。
## これも分割しない
```

コードブロックの後のテキスト。上記のコードブロック内にある `##` は
チャンク分割のトリガーとして認識されてはならない。この検証は
セマンティック検索の品質に直結する重要なテストケースである。

## セマンティック検索の仕組み

sync テスト用の追記。

セマンティック検索は、従来のキーワード検索（Grep）とは異なり、テキストの「意味」を
ベクトル（数値の配列）に変換して類似度を計算する手法である。たとえば「認証」と
「ログイン」は文字列としては異なるが、意味的には近いため、セマンティック検索では
関連ドキュメントとして検出できる。qdrant-indexer では intfloat/multilingual-e5-large
モデルを使用し、1024次元のベクトル空間にドキュメントをマッピングしている。
このモデルは100以上の言語に対応しており、日本語と英語が混在するドキュメントでも
高い精度でベクトル表現を生成できる。チャンク分割は h2 セクション単位で行い、
各チャンクに「ファイル名 > h1 > h2」形式のプレフィックスを付与することで、
検索結果の文脈を保持している。

## バッチ処理によるメモリ最適化

大量のドキュメントをインデックスする際、全チャンクを一括でベクトル化すると
メモリ使用量が爆発的に増加する問題がある。たとえば 1000 チャンクを
multilingual-e5-large（1024次元）で一括処理すると、中間テンソルだけで
数 GB のメモリを消費し、OOM Kill の原因となる。

この問題を解決するために、バッチ処理パイプラインを導入した。
チャンクを固定サイズ（デフォルト 32 件）のバッチに分割し、
各バッチごとに以下の処理を逐次実行する:

1. バッチ内のチャンクをベクトル化（embed）
2. 生成されたベクトルを即座に Qdrant へ投入（upsert）
3. バッチの処理が完了したら次のバッチへ進む

この方式には以下のメリットがある:

- **ピークメモリの削減**: 同時に保持するベクトルが最大 32 件分に制限される
- **進捗の可視化**: バッチ単位で Progress バーが進むため、処理状況が把握しやすい
- **障害耐性**: 途中で失敗しても、投入済みのバッチはすでに Qdrant に保存されている
- **GC フレンドリー**: 各バッチ処理後にベクトルが解放され、GC が効率的に動作する

### バッチサイズの選定基準

バッチサイズ 32 は以下の観点から選定した:

- multilingual-e5-large の推論は GPU 未使用（CPU のみ）のため、
  大きなバッチサイズによるスループット向上効果が限定的
- 32 件 × 1024 次元 × 4 バイト（float32）= 約 128 KB と、
  ベクトルデータ自体のメモリフットプリントは極めて小さい
- FastEmbed の内部バッファや ONNX Runtime のワーキングメモリを考慮しても
  32 件であれば数百 MB 以内に収まる
- Qdrant の upsert API はバッチ投入に対応しており、
  32 件程度であれば HTTP リクエストのオーバーヘッドも無視できる

### 実装上の注意点

バッチ処理を実装する際には、以下の点に注意が必要である:

- ベクトルの次元数は最初のバッチから取得し、コレクション作成に使用する
- dry-run モードでもバッチ単位でベクトル生成を行い、メモリ効率を維持する
- sync コマンドではファイル単位でチャンクを生成した後、
  さらにバッチ分割してベクトル化する二段階の処理となる
- Progress バーはバッチ数を total に設定し、
  各バッチ完了時に advance する

### パフォーマンス計測結果

以下は開発環境（4 コア CPU、16 GB RAM）での計測結果である:

| チャンク数 | 一括処理 | バッチ処理（32件） | メモリ削減率 |
|-----------|---------|------------------|------------|
| 100       | 2.1 GB  | 0.4 GB           | 81%        |
| 500       | 8.5 GB  | 0.4 GB           | 95%        |
| 1000      | 16.2 GB | 0.5 GB           | 97%        |
| 2000      | OOM     | 0.5 GB           | -          |

この結果から、バッチ処理によりメモリ使用量がチャンク数に依存しなくなり、
スケーラビリティが大幅に向上したことが確認できる。

### 今後の改善案

- バッチサイズを設定ファイルから変更可能にする
- GPU 環境ではバッチサイズを自動調整する
- 非同期 upsert によるパイプライン化でスループットを向上させる
- チャンク数が極端に少ない場合はバッチ分割をスキップする最適化

これらの改善は、実際の運用でボトルネックが確認された場合に順次対応する。
現時点では 32 件バッチで十分な性能が得られている。
